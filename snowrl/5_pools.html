Options

AssetCache<T> の実装案としては:

  1. アセットを Arc<Mutex<T>> に入れる → 採用

    TODO: serde の際に重複したオブジェクトを作らない (interning) ように気をつけます。

  2. アセットをプールに入れる

    1. Handle<T> を弱参照にする (例: thunderdome)

    2. Handle<T> を強参照にする (例: froggy, bevy_asset)

      Handle<T> の参照カウントに必要なのは drop を通知する仕組みです。これは一方的な通信なので、 Arc<Mutex<T>> ではなく mpsc, crossbeam または flume を使用できます。どちらが良いのかは分かりませんが……

Arena<T> は、 Vec<T> の wrapper です。なぜ生の Vec<T> ではダメなのかは、 generational_arena に書いてあります。そもそも、 Vec<T> のアイテムを 削除 すると、他アイテムの位置がズレます。

3.1. Layer

Sprite は layer → z 軸の順でソート予定です。ひとまず Layer のみを考えます:

pub struct Layer {
    pub nodes: Pool<Node>, (1)
    pub anims: Arena<Anim>, (2)
}
1 Node は Handle の数がゼロになるまで消滅しません。たとえば、 Node を参照するアニメが生きている限り、 Node は死にません。
2 アニメは、再生が終わったら自動的に削除されます。

Node が Pool に入っている都合上、 Pool 内 Node への WeakHandle をソートする形になると思います。

3.2. Node

Node は決め打ちです:

pub struct Node {
    draw: Draw,
    params: DrawParams, (1)
}

pub enum Draw {
    Sprite(SpriteData),
    NineSlice(NineSliceSprite),
}

pub enum DrawParams {
    pub pos: Vec2f, (2)
    pub size: Vec2f,
    pub rot: f32,
    pub color: Color,
    pub scales: Vec2f,
    pub origin: Vec2f,
}
1 アニメは全 Node 共通の DrawParams を書き換える形で働きます。
2 未だに独自のベクトル型を使っています。

3.3. Anim (tween)

Fixed set of animations

enum_dispatch で Trait をアップキャストしたような enum を作ります:

#[enum_dispatch]
pub trait AnimImpl {
    fn tick(&mut self, dt: Duration);
    fn is_end(&self) -> bool;
}

pub struct PosTween { .. }
impl AnimImpl for PosTween { .. }

pub struct ColorTween { .. }
impl AnimImpl for ColorTween { .. }

pub struct AlphaTween { .. }
impl AnimImpl for AlphaTween { .. }

/// One of the [`AnimImpl`] impls
#[enum_dispatch(AnimImpl)]
pub enum Anim {
    PosTween,
    ColorTween,
    AlphaTween,
}

Tween builder (animation builder)

Builder を用意して、最低限の API が整いました:

タイトルロゴのアニメ
anims
    .builder()
    .node(&nodes.logo)
