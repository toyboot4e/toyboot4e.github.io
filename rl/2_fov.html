2. How to calculate FoV

円形視野の計算方法を考えます。

2.1. Circle = octants

円形視野の計算を、 45 度ずつの計算 8 回に分けます。 8 等分された円の部分を八分円 (Octant) と呼びます。

Octant ごとに列座標系を取ると、座標成分は (column, row) と表すことができます。たとえば、 (column, row) = (1, 3) は、 1 番目の縦列と 3 番目の横列の交点です。

基底ベクトルを用いて、

\[(x, y) = column \cdot \mathbb{e}_{column} + row \cdot \mathbb{e}_{row}\]

したがって、 Octant に対して視野計算ができるなら、基底ベクトルを差し替えていくことで、円全体の視野を得ることができます:

Figure 3. Octants A, B, .., H and unit vectors
図では、 y 軸を下向きに取っています。

2.2. Scanning rows

Octant.B の場合で、視野の計算を考えてみます。傾きを \(slope := column / row\) と定義します。 1 つの Octant に対応するセルは、 \(0 \leq slope \leq 1\) の範囲に収まります。

Figure 4. Field of view in Octant.B

ゲームに必要なのはセル単位の視野です。そこで、可視範囲が セルの中心 を含むとき、そのセルを見ることができるとします:

さらに考えを進めます。視野の計算は、 row の走査に分けることができます:

縦向きの矢印は、走査 (スキャン) を表しています。スキャンは、傾きの範囲内のセルに対して行われ、スキャンされたセルは視野内にあるとします。

2.4. Edge cases (□ model vs ◇ model)

これまでは、壁の形を□形として扱ってきました。ここで、角における視野を考えます。□形に基づくと、視野は直線になります:

FoV (□ block model0
+@#...    @ player              +@++++
+#+...    # opaque cell         +#++++
...+..    . transparent cell    ...+++
....+.    + visible cell        ....++
.....+                          .....+

しかし、もしもこれが『自然ではない』と考えるなら、壁の形を 45 度回転させ、トランプのダイヤマークのような判定◇を使用します。すると、より permissive (寛容) な視野が得られます:

FoV (◇ block model)
+@#...    @ player              +@++++
+#++..    # opaque cell         +#++++
..++++    . transparent cell    ..++++
...+++    + visible cell        ...+++
....++                          ....++
検討項目は、視野の対称性です。モンスターの視野が主人公の視野よりも狭い場合は、少しゲームが簡単になります。なるべく『フェア』な式を使いたいところですが、多少のエラーはゲームプレイに影響しないので、見栄えさえ良くできればと思います。

2.5. One more fix

セルの中心を可視判定に用いました。しかし、 中心は隠れているけれど、その角 (かど) が視野角に影響するはずの壁 が出てきます。それらの当たり判定を考慮して、走査範囲を更新する必要があります。

4. Visualizing FoV

4.1. .. is, rendering shadows

描くのは影です。規格化距離 (距離 / 視野の半径) から影の濃さを定義します:

\[shadowAlpha = maxShadowAlpha * f(\frac {distance} {radius})\]

まず \(f\) を 1 次関数にしましたが、明るさの変化が滑らか過ぎて、セルとセルの境界がはっきりとしませんでした。グリッドベースのゲームにおいては、機能的とは言えません。

代わりの 補完関数 を用います。最も見栄えが良かったのは CubicInOut でした。

記事の画像では、常に CubicInOut を補完に使用しています。
