2. How to calculate FoV

ある点から、円形の視野を計算する方法を考えます。

2.1. Circle = octants

円形視野の計算を、 45 度分の計算 8 回に分けます。このとき、 45 度分の円の部分を、 Octant と呼びます。

Octant ごとに列座標系を取ると、座標成分は (column, row) と表すことができます。たとえば、 (column, row) = (1, 3) は、 1 番目の縦列と 3 番目の横列の交点です。

値としての column (縦列) は、『縦列番号』とも考えられます。 row は横列番号です。

基底ベクトルを用いて、

\[(x, y) = column \times columnUnitVec + row \times rowUnitVec\]

したがって、 Octant に対して視野計算ができるなら、基底ベクトルを差し替えていくことで、円全体の視野を得ることができます。

Figure 2. Octants and their names [1]

図では、 y 軸を下向きに取っています。

2.2. Scanning rows

2.4. Edge cases (□ model vs ◇ model)

これまでは、壁の形を□形として扱ってきました。ここで、角における視野を考えます。□形に基づくと、視野は直線になります:

FoV (□ block model0
+@#...    @ player              +@++++
+#+...    # opaque cell         +#++++
...+..    . transparent cell    ...+++
....+.    + visible cell        ....++
.....+                          .....+

しかし、もしもこれが『自然ではない』と考えるなら、壁の形を 45 度回転させ、トランプのダイヤマークのような判定◇を使用します。すると、より permissive (寛容) な視野が得られます:

FoV (◇ block model)
+@#...    @ player              +@++++
+#++..    # opaque cell         +#++++
..++++    . transparent cell    ..++++
...+++    + visible cell        ...+++
....++                          ....++
検討項目は、視野の対称性です。モンスターの視野が主人公の視野よりも狭い場合は、少しゲームが簡単になります。なるべく『フェア』な式を使いたいところですが、多少のエラーはゲームプレイに影響しないので、見栄えさえ良くできればと思います。

2.5. One more fix

セルの中心を可視判定に用いました。しかし、 中心は隠れているけれど、その角 (かど) が視野角に影響するはずの壁 が出てきます。それらの当たり判定を考慮して、走査範囲を更新する必要があります。

Scanning a row

列を走査します。 最後の図 で解説した通り、状態を上手く使います。傾き計算の Rule クラスは後で解説します。

OctantScanner
enum RowScanState {
    Initial,
    WasOpaque,
    WasTransparent,
}

// Returns true if finished
bool scanRow(int row, ScanContext cx) {
    var rowVec = this.rowUnit * row;
    (int fromCol, int toCol) = Rule.colRangeForRow(row, cx.radius, this.startSlope, this.endSlope);

    // finish the scan if the view is completely blocked
    if (toCol < fromCol) return true;

    { // finish the scan if the row is out of the map
        var initPos = cx.localToWorld(rowVec);
        if (!cx.map.contains(initPos.x, initPos.y)) return true;
    }
    // ~~

これが肝心の走査です:

    // scanning through the row
    var state = RowScanState.Initial;
    for (int col = fromCol; col <= toCol; col++) {
        var pos = cx.localToWorld(rowVec + this.colUnit * col);

        // skip points out of the map
        if (!cx.map.contains(pos.x, pos.y)) return false;

        // scan the cell
        if (cx.map.isOpaeue(pos.x, pos.y)) { (1)
            if (state == RowScanState.WasTransparent) { (2)
                this.splitScan(Rule.updateEndSlope(col, row)).scanOctant(cx, row + 1);
            }
            state = RowScanState.WasOpaque;
        } else { (3)
            if (state == RowScanState.WasOpaque) { (4)
                this.startSlope = Rule.updateStartSlope(col, row);
            }
            state = RowScanState.WasTransparent;
        }

        cx.fov.light(pos.x, pos.y); (5)
    }

    return state == RowScanState.WasOpaque;
}

OctantScanner splitScan(float endSlope) {
    return new OctantScanner(this.rowUnit, this.colUnit, this.startSlope, endSlope);
}
1 不透明なセルを見つけた場合、
2 直前に走査したセルが透明だったなら、スキャンを分割します。
3 透明なセルを見つけた場合、
4 直前に走査したセルが不透明だったなら、 startSlope を更新します。
5 どのケースでも、走査したセルは可視化します。

Slope calculations

Rule クラスは式の集まりです。ディテールに影響します。

static class Rule {
    // Range of columns in a row to scan through; [from, to]
    public static(int, int) colRangeForRow(int row, int radius, float startSlope, float endSlope) {
        int from = (int) Math.Ceiling(startSlope * row); (1)
        int to = Math.Min(
            (int) Math.Floor(endSlope * row), (1)
            Rule.maxColForRow(row, radius) (2)
        );
        return (from, to);
    }

    public static int maxColForRow(int row, int radius) {
        // we add 0.5 to the radius so that FoV looks more natural
        return (int) Math.Sqrt((radius + 0.5) * (radius + 0.5) - row * row); (2)
    }
1 傾き範囲がセルの中心を通るセルを走査範囲とします。 Math.Ceiling は切り上げ、 Math.Floor は切り捨てをします。
2 走査範囲を円形にします。この補正無しだと、正方形範囲の走査となります。半径に 0.5 を加える理由は、円が自然な形になるためです (c.f. Circle fill on a grid) 。

傾き範囲の更新は、不可視セルのモデル (Edge cases (□ model vs ◇ model)) に応じて式を変えます。

□ block model

セルの形を□形だとみなした場合、以下のような式となります:

□ rule
    // Called after scanning opaque cells
    public static float updateStartSlope(int col, int row) =>
        Math.Min(1f,  (col + 0.5f) / (row - 0.5f)); (1)
    // Called when splitting a scan
    public static float updateEndSlope(int col, int row) =>
        (col - 0.5f) / (row + 0.5f); (1)
1 傾き範囲をセルの角に対して更新するため、半セル単位で計算します。 [4]
◇ block model

セルの形を◇形だとみなした場合、以下のような式となります:

◇ rule
    // Called after scanning opaque cells
    public static float updateStartSlope(int col, int row) =>
        (col + 0.5f) / row;
    // Called when splitting a scan
    public static float updateEndSlope(int col, int row) =>
        (col - 0.5f) / row;
式を差し替え可能にするなら、 Rule を静的クラスではなくオブジェクトにします。 Strategy パタン？

Permissive な column は、以下のように計算しました。なぜか複雑な式を使っていますが、普通に切り上げたら良かったと思います。

Rule
// Used to detect a cell whose center it not in the range
// but one of whose vertex may hide following cells
public static int colForSlopePermissive(float slope, int row) {
    return Math.Min(
        Rule.maxColForRow(row, radius),
        (int) Math.Ceiling(slope * (row + 0.5) - 0.500001f)
    );
    // We reduced (0.5f + small_amount) not to include a vertex
    // which is on an `endSlope` e.g. (row, col) = (1, 1),
}

4. Visualizing FoV

視野の可視化を試みます。綺麗な画面を作りこめるのが、グラフィカルなゲームを作るときの特権だと思います。

不透明度を含む renderig が上手くいかない場合は、アルファ値の計算を間違えている可能性があります。アルファ値の計算法 (blend mode) には、 straight alpha と premultiplied alpha があり、使っているフレームワーク（の設定）にあった計算が必要です。

4.1. Rendering

まずは、セル単位で明度を調整する方法を考えます。

Shadow/light casting

セル毎に 影を 描くことにしました。

Light casting

以下の二案はボツとしました。

  • まずマップ全体に影を描いてから、視野範囲を明るくする方法が考えられます。しかし、 Color の RGB 値は負の数を取れないため、影を描いたときに正確な色情報が失われる恐れがありました。

  • もう 1 つ考えられるのは、影の Texture を作ってから、出力画面に合成する方法です。が、やり方が分からなくて止めました。負荷が増えそうな気もします。

How to render

僕は Renderable の render メソッドを実装しました。他に思いつくのは、 Sprite をセル毎に配置する方法ですが、本質的にはどちらも同じです。

  • Sprite を設置した場合は、 Renderer がそれらを自動的に走査して、セルの明るさに影響を与えます。

  • render メソッドを実装する場合は、メソッド中でセルを走査して、明るさを調整します。

Immediate mode か retained mode かという差です

後は、状態が必要なときに、 Sprite オブジェクトに入れるか、自前のフィールドに入れるかという違いがあります。

Decaying shadows

影の濃さを定義する関数 \(f\) を考えます。引数は規格化距離です:

\[dimAlpha = maxDimAlpha * f(distance / radius)\]

まず線形の場合ですが、明るさの変化が滑らか過ぎて、セルとセルの境界がはっきりとしませんでした。グリッドベースのゲームにおいては、機能的とは言えません。

代わりの 補完関数 を用います。まず sin カーブが思い浮かびましたが、最も見栄えが良かったのは CubicInOut でした。

記事の画像では、常に CubicInOut を補完に使用しています。

4.2. Animation

視野が更新されたとき、セルの明度を滑らかに遷移させたいと思います。

何らかの方法で、 FoV の変更を追跡する必要があります。また、追跡のためのコストは、セル毎に \(O(1)\) であってほしいと思います。

Double buffer of FoV

プレイヤの視野データを 2 つ持ち、交互に更新します。すると、現在の視野データと、前回の視野データが手に入り、セル毎に前回と今回の明るさが分かります。後は経過時間を参照すれば、滑らかに明度を推移させることができます。

シンプルだったため採用しました。

この方法は、表現力に限界があります。キャラの移動が視野アニメーションの更新よりも速い場合は、違和感が出ると思います。キューに視野データのコピーを入れていく……？
Ways not in use

  • Hash table of changes

明度の変更を (x, y) の hash table に入れます。メモリや状態の管理が大変そうなので止めました。

  • Grid of states

セル毎にアニメーション状態を持たせる方法です。 Sprite を配置するのとほぼ等価であり、最強ですが、ボツとしました。

public class FovViewStateForCell {
    public float previousAlpha;
    public float targetAlpha;
}

マップ内の全セルに持たせると、無駄が多いです。あるいは、カメラ周辺のセルにのみアニメ状態を持たせることもできますが、複雑になるため止めました。
