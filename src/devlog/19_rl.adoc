==== Drop

`Drop` は引数を取りません。オブジェクトが `Drop` でメモリを開放するためには、そのフィールドにメモリ解放のための依存性を入れておく必要があります。しかし、 Rust の borrow rules に従うと、 `&mut` は共有できません (基本的に) 。したがって、 `Rc<*mut Device>` のようなフィールドを作るのは、やや反則的 footnote:[単スレッドなら `Rc` で `Device` を共有して良いのではないかと思います。が、そもそも borrow rules が何のための物であるかを理解する必要がありそうです。マルチスレッドのため？？　マルチスレッドに関しては、 Vulkan の wrapper (https://github.com/vulkano-rs/vulkano[Vulkano]) が `Arc<Device>` をフィールドに持っているらしく、参考になるのかもしれません。] に思えます。考え中……

. 引数付きの `Drop` (いわば `DropWith<T>`) を用意して、
. 親オブジェクトがリソース破棄に責任を持つならば、所有権の問題は無かったことにできそうです。特に pool を使っているなら簡単そうだと思います。

NOTE: 補足: graphics API では、主要な API が `Device` と名付けられています。 `Device` は GPU 側の操作全般に要求され、 `Texture` などのリソースの生成・破棄にも必要です。 `Device` は不透明 (フィールドが見えない) オブジェクトとして提供されており、ユーザは `Device` のポインタ (`*mut Device`) を使って graphics API の機能を利用できます。 Rust のポインタは、 `unsafe` なデリファレンスを経由して `&mut` に変換できます。

==== Deserialize

リソース管理 footnote:[どうでもいいことですが…… asset 管理型は `Ass` とか `AssSet` とか `AssSets` みたいな下品な名前にしようと思います。] のモデルは、主に 2 つと思います:

. `Pool<T>`: pool + handler
+
アセットの所有者は `Pool` です。アセットのユーザは、リソースへの `Handler` (もしくは `Handler` を得るための `Id`) をフィールドに入れます。

=== Multi-threaded rendering

FNA3D では、マルチスレッドは基本的に考えられていません。 `Device` は `Rc` を使って

.graphics_api.rs
[source,rust]
----
pub struct Device { inner: Rc<DeviceDrop> }
pub struct DeviceDrop { raw: *mut ffi::Device }
impl Drop for DeviceDrop { .. }
----

これで、 GPU 側のオブジェクトの破棄を自動化できます:

.texture.rs
[source,rust]
----
pub struct Texture2dDrop {
    device: Device,
    raw: *mut ffi::Texture,
    w: u32,
    h: u32,
}

impl Drop for Texture2dDrop {
    fn drop(&mut self) {
       self.device.add_dispose_texture(self.raw);
    }
}
----

しかし、 Vulkan などでは、マルチスレッドを考慮している _らしい_ ではないですか。したがって、こんなコードは通用しなくなるのではないか、という不安がありました。
