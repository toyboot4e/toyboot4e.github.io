制作中フレームワーク ANF で Pong を作りました。

1.1. Pong

Pong とは、画面端のバー (paddle) でピンポン球を打ち合うゲームです。

まずは paddle を表示しました:

おっと、イカちゃんが paddle 役をやってくれるようです！　画面に調和を感じますね。

>>> Sub textures enabled

ika-chan.png は 12 のイカちゃんが詰まった画像ファイルなので、左右の paddle は、元 texture の sub texture と言えます。

Delta time

左のパドルをプレイヤが動かせるようにしたいと思います。パドル位置は \(\mathbb{x} = \int \mathbb{v}(t) dt\) という積分値ですが、オイラー法によって 1 次式に近似できます:

\[\mathbb{x}(t+dt) \sim \mathbb{x}(t) + \mathbb{v}(t) * dt \\ \mathbb{v}(t+dt) \sim \mathbb{v}(t) + \mathbb{a}(t) * dt\]

ここで \(dt\) を与えるために、 FNA の ゲームループ をパクりました。写経は学びと実装を両立できて効率良いですね。

これで、イカちゃんを操作できるようになるはずです:

ンンン　画面外に行ってしまいました。画面端で止まるように修正します。

Bounce

画面端や paddle に当たったボールには反射してもらいます。

おっと 反射してくれません。イカちゃん (paddle) の下端に当たり判定が無かったようです。

2.1. Tips on Rust

API guideline

Rust APIガイドライン を読む。読みます。

翻訳グループの仕事は素晴らしいですね。

Nightly version of cargo doc

Nightly 版のドキュメントでは、アイテムのリンクが楽です:

//! `cargo +nightly doc` allows these link notations:
//!
//! [`X`] and [`A`](crate::a::A)

struct X {}

mod a {
    struct A {}
}

crates.io も、 cargo doc は nightly 版を使用しています。

2.2. build.rs (build script)

ANF フレームワークのベースには、 Rust-FNA3D があります。これは、 C ライブラリ FNA3D のラッパーです。 Rust-FNA3D を crates.io に出す前に、以下を自動化したいです:

  • FNA3D が無ければ pull

  • FNA3D の CMakeList.txt にパッチを適用

  • FNA3D がビルドされていなければ、 cmake で自動ビルド

  • Windows でも動く build.rs にする

2.3. Global objects

グローバルなデータの扱い方には、 3 通りあると思います。

1. Big Context object

ggez や tetra のやり方です。ウィンドウ、画像、音声などを Context 入れておきます。フレームワークの関数は Context オブジェクトを受け取って機能を発揮します:

ggez::graphics::draw(Context, DrawParams)

  • この方法では、 Context を引数として渡して回ることになります。すると、オブジェクト指向な API を作れないと思います。それはそれで、コンスタントな API となって良いかもしれません。

    細部が分からないので、 ggez や tetra の上に作られたフレームワークを見てみたいです。

  • macroquad では、 Context オブジェクトはユーザから隠されており、グローバルオブジェクトとしてフレームワーク内で共有されています。

2. Dynamic map

Bevy ESC の Resource がこれだと思います うーん？ HashMap<Type, Box<dyn Any>> 相当のものを使います。

3. Subclass

XNA では、 Game クラスを継承することで、フレームワークを拡張できます。多態性無しでも継承は結構いいんじゃないかと思いました。

ANF では継承をイメージして、 Game クラス相当のものをサンプルコードとして提供しようと思います。ユーザは、そのコードを書き換えることでフレームワークを拡張します。継承すべきコードをユーザ側のコードに入れてしまう形です。
