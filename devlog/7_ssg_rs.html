Apr 17, 2020

1. About

ブログ生成を bash から Rust に移行しました。

  • repository / ssg_rs 削除済み

html を木にパースしてから加工しています。 Handlebars の使い方を調べるか、 .adoc ファイルをテンプレート代わりに使う方向性の方が良かったかもしれません。

1.1. Background

前回、 bash を使ってブログを生成しました。十分に機能しましたが、機能追加には厳しいものがありました。したがって、今回はプログラミング言語を使い、柔軟性を高めました。

2. impl ssg_rs

2.1. Environment

rvm を使って Asciidoctor を再インストールすると、原稿の変換速度が 3 倍以上 に改善されました。Mac の Homebrew/rbenv 経由だと、間のスクリプトが遅かったのではないかと思います。 rbenv を使っている人は、 検討 されてみてはいかがでしょうか。

また、 Rust では rust-analyzer が rls の代替となるようなので、移行しました。

2.2. Parsing

html を文字列ベースで (タグ名を enum にせずに) パースしてみました。また、 html の builder を作りました。しかし、結局必要だったのは templating engine に相当する機能だと分かりました。 つまり無駄でした このブログ専用のジェネレータとなってしまいました。

2.3. How to use

ssg_rs は、以下のようなファイル構成を前提とします:

Directory layout for ssg_rs
.
├── img/
├── posts/
├── drafts/
├── site/
├── ssg/
└── styles/

ssg/ 以下に、 Rusty Object Notation で書いた設定ファイルを置きます:

ssg/setup.ron
Config(
    layout: Layout(
        markups: ["posts", "read"],
        dest: "site",
        img: "img",
        styles: "styles",
    ),
    site: Site(
        title: "Toy toy beam",
        message: "toyboot4e のブログです。プライベートな気分で記事を書きます。",
    )
)

struct の上に #derive[Deserialize] と書くだけで、設定ファイルから deserialize できるようになります。

2.4. Rust!

CLI 引数は clap (ver. 3) でパースしました。また、標準ライブラリがしっかりと活躍しました。 エラーが大量に出る のが Rust の特徴だと感じましたが、かえって インクリメンタルに 変更に強いコードへリファクタリングする必要が出て良かったです。

エラーが出る度に、共通箇所をまとめました。その結果、高階関数が非常によく活躍しました。共通の引数が多い場合は、 struct にキャプチャすべき変数をまとめ、関数はメソッドになりました。

3. Wrapping up

bash (400 行弱) から Rust (2,000 行強) へ移行しました。この程度の複雑さには、コンスタントに対処できそうです。成果としては、オプションを指定することで、下書きファイル (drafts/*) も変換できるようになりました。

今後は、 より多彩なサイト生成が可能な SSG を目指してみようと思います。設計的に面白そうですし、 Asciidoctor に native 対応した SSG は少ないこともあり、若干のモチベーションがあります。差し当たり、 mdbook の代替として、 adbook を作ることを目標に考えています。

⇒ 追記: マークアップ言語と SSG の両方を自作する方向に舵を切りました。マークアップ言語にテンプレート言語を兼ねさせてみたいです。
