1. Summary

Asciidoctor を使って、本ブログの html を作成しました。また html/css に入門しました。

  • スクリプト、原稿ファイル

  • 生成された html

既に、この記事のスクリプトはブログ生成に使用していません。ブログの見た目も変化しています。

やり方としては、 Asciidoctor で原稿を html に変換し、 bash でヘッダとフッタを追加します。一人で作業する分には、シンプルで悪くないと思います。

2. Static site generation

始めに、最もシンプルな SSG として、 mdbook を検討しました。これは Rust 言語の 本 に使用されており、見た目も良いと思います。しかし、 画像に caption を加えられない 、など markdown の表現力に制限を感じました。したがって、他の markup 言語が必要でした。

2.1. Markup language

様々な言語が見つかりました。ツールの 選択に迷った時は、ドキュメントが良い環境を選ぶ ことにしています。 AsciiDoc (処理系は Asciidoctor) に行き着きました。

AsciiDoc には、 macro, block, attribute といった 一般的な構文 が用意されており、豊富な機能にアクセスできます [1]:

Syntax example (macros and blocks) 
// macros
image::path[align="center"]
kbd:[Ctrl+z]
footnote:[脚注]

// blocks with attributes
[source,csharp]
----
public class TestClass {
    // code block
}
----

[quote]
____
Shall we dance! // quoting block
____
Demo

とりわけブログに良かった機能は、 callouts や footnote[2] (脚注) でした。脚注は今出した通りのもので、 callouts は、次のように表示できます:

Example of callouts
$ asciidoctor devlog.adoc (1)
1 これが callout です。
\[数式も書けます \\ convert (adoc) := asciidoctor(adoc, extensions) \\ where\ extensions\ are\ awesome\]

Enter, Ctrl+T ← キーボードキーです [3]。

sciidoctor-diagram のおかげで、文字データから図を作り出すツールも使用できます。

  • マインドマップを PlantUML で描いてみました [4]:

    箇条書きで書けるのが便利です。上下にノードを伸ばせないのがイマイチです。

  • ditaa で図も書けます:

    Figure 1. Separation in serde (maybe it’s like this)
もう 1 点優れているのは、 本の作成が念頭に置かれている ことです。 AsciiDoc では、分割ファイルの include ができます。

これで、 記事部分は思い通り作れます 。後は外枠を補填し、ページとページをリンクで繋げることでサイトとします。

2.2. Generating html

Asciidoctor を使ったブログ生成の話題は、 SSG (static site generator) に限られていました。大半の SSG には、メタデータを yaml frontmatter として書かされる ようです。しかし、それでは AsciiDoc 本来の書き味を損うと捉え、 自力で html を加工 することにしました。

Recommended way?

Asciidoctor のツールチェインには、 Ruby で書ける 拡張 や stylesheet factory がありました。実際、 上手く使った人 もいます。しかし、使い方を理解するのが大変そうだったので、今回はパスしました。

.adoc → .html → .html

Asciidoctor で .adoc ファイルを変換すると、記事部分だけのシンプルな .html が出力されます。それに bash で header, footer を追加しました (ソース)。

ただ、何でも bash でやるのは良くないですね。プログラミング言語を使うか、 html の加工にはテンプレート言語を使った方が良かったと思います。

Hand-written css

ターミナル画面の再現 をコンセプトに、 css に着手しました。素人ではあったものの、インターンシップで html/css の経験があったため、概要は理解していました。

まず記事を中央に配置し、ターミナルの配色をコピーしました。それだけで、悪くない画面になりました。その後修正を繰り返し、おそらく細部に問題がありますが、ブログとしては十分だと思います。満足しています。

3. Wrapping up

以前はサイトの作り方など知らず、大きな壁に思えたものですが、どうやら html のファイル構成 ~ サイト と思って良さそうです。『難しいはずの部分』は、 GitHub Pages やブラウザ任せにできました。

Asciidoctor は得るべき道具だったと思います。今後も、 サイト・レポート・スライドの作成 に使っていこうと思います。

Appendix

A. Markup languages other than AsciiDoc

org-mode にも注目しています。まだ調べている最中ですが、この選択は何らかの trade-off になるという印象です。それだけ Asciidoctor の完成度が高いのだと捉えました。

両言語の表現力は、ほぼ同等に見えます。ただし、 org-mode は UI とツールが非常に発達 しています。特に Emacs から使うと強力で、 TODO 管理などに向いているようです。任意の設定ファイルを .org ファイルに書き込んで、コマンド 1 つで実ファイルにコピーする、みたいな使い方をしている人もいます。この拡張性は Emacs + Emacs Lisp に由来しています。

B. Asciidoctor for other than html

ブラウザの印刷機能を使えば、 html から pdf を生成できます。これがレポートの作成に重宝しました。というのも、 AsciiDoc では 分割ファイルの include が可能 なため、長い原稿も編集が楽だからです。 Office はもう要らないですね。

拡張機能が豊富なのも、 Asciidoctor の特徴です。 pdf や スライド[5]、電子書籍も作れるようです。実際、 ProGit の執筆に使用された という実績があります。

ただし、 html 以外の出力は、スムーズに行かないと思います。 asciidoctor-pdf は、デフォルトのフォントが日本語を含まないなど、様々な問題が出ます。

C. Other ways to convert html files

bash で html を加工する必要は無かったはずです。他の方法を検討してみます。

Editing an html as a tree

Asciidoctor が作った html を木としてパースし、加工してから、再び html として出力します。これは意外と複雑そうです。

Templating engines

ページ外観の html を用意し、そこに記事内容をはめ込みます。これは templating と呼ばれるそうです。

Pseudo html template
<!DOCTYPE html>
<html lang="jp">
<head>
    <!-- 省略 -->
</head>
<body>
    <!-- templates --!>
    {{header}}      <!-- パンくずリスト --!>
    {{article}}     <!-- 記事 --!>
    {{footer}}      <!-- footer !-->
</body>
</html>

pandoc や handlebars が『それっぽい』ツールの気がします。ただし、構文がキメラっぽく見えて苦手に思いました。今回は使いませんでした。

D. Getting started with html/css

今回、最も難しかったのは、 html/css を理解することでした。入門書こそ多いのですが、ポイントを掴みにくい仕組みだと思います。

Tips

たとえば、こういった情報を集めるのに手間がかかります。

  • 常識

    • 静的サイトの URL は、ファイル構成と一致 します。ルートとなるページは index.html で、 path/to/file.html が 1 ページに相当します。

    • ルートに 404.html を置くと、存在しないページにアクセスしたときに表示されます (GitHub Pages の場合) 。

    • 自分のコンピュータ上で絶対パス (/some/url という形式で書く、サイトのルートからの相対パス) を使うには、 ローカルサーバ を使います。

    • ブラウザ毎に、デフォルトの CSS 設定があります（余白など）。無効化する方法として、 Reset CSS が有名ですが、賛否両論あります。

  • 操作

    • Chrome/Firefox で Cmd+Shift+c を押すと、マウスホバーでレイアウトを確認できます。

  • 手法

    • このサイトのトップは、 flexbox という機能を使ってレイアウトしています。

      トップ画面は変更しました。要素を横に並べるときに flex は便利です。

    • フォントは Google Fonts から取ってくるのが楽ですが、デメリットもあるようです。

    • MathJax (数式表示) や Prism.js (構文ハイライト) が便利な JS だと思います。これらは <script> タグを追加することで使用できます。 JS ファイルは、ダウンロードするかネット上のものにリンクします。

ブラウザの仕組みが気になったときは、 ブラウザの仕組み: 最新ウェブブラウザの内部構造 などが参考になります。意外とテキストエディタに似ているかもしれません。

Layout with CSS

簡単に html/css の概要を説明してみようと思います。

まず、ミニマルな html は、次のような構造を持ちます。

Minimal html
<!DOCTYPE html>
<html lang="en">
  <head>
    <title>The website</title>
    <link rel="stylesheet" href="style.css">
  </head>
  <body>
    <!-- page content -->
  </body>
</html>

木構造としては以下のような関係と言えます。

html
├── head
│   ├── title
│   └── link
└── body

まず、 body の下にデータを足していきます。 article を配置し、その下に header, content, footer を作ったとします（手書きで、あるいは markdown などから）。

Basic html
  <body>
    <div id="article">
        <div id="header">
            <!-- content !-->
        </div>
        <div id="content">
            <!-- content !-->
        </div>
        <div id="footer">
            <!-- content !-->
        </div>
    </div>
  </body>

次はレイアウトです。ブラウザは、 css を元に、 html の木を表示します。 css は、 cascading style sheet というだけあって、親要素から子要素へと情報を受け継いで行きます。

そこで、 article の横幅を 40rem (40 文字) くらいに設定すれば、その下に配置された header, content, footer の横幅も制限され、 行あたり文字数を設定できます 。

さらに、 article の margin を auto にします。すると、記事を画面中央に配置できます。このブログの基本的なレイアウトは、実にそれだけです。

Basic layout
div#content, div#footnotes>*{
    max-width:44em;
    margin:auto;
}

基本ができたら、後は自力で何とかできると思います。たとえば、画面脇に目次を表示したいとします。ひとまず、 Fixed Sidebar を真似れば表示できます。さらにメニューを、横に狭い画面では非表示にするとして、たとえば media query で responsive にすることを検討します。この段階まで来たら、『静的サイトを作ることができる』と言って良いのではないでしょうか。

もっとも、『正しい方法』『質の高い html/css 』などを求め始めると、格段に難度は上がりそうです。僕なら mdbook を見てみますが、極めるのは非常に大変そうですね。趣味で良かったです。
